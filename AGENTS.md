# 协作说明

本仓库用于维护中文考研数学 Obsidian 知识库。后续协作者或智能代理在修改仓库时，请遵守以下约定。

## 定位

- Obsidian 是主要阅读、复习和知识组织环境。
- GitHub 只用于备份、版本管理和协作记录。
- Codex 用于批量整理、格式修复、生成题目和维护仓库。
- `scripts/`、`problems/`、`tests/` 是辅助工具，不作为主复习入口。

## 内容约定

- 所有面向学习者的说明、题目、解析和文档使用中文。
- 优先保证 Obsidian 阅读体验，笔记应适合直接复习，而不是写成代码项目说明。
- 使用中文文件名和中文笔记标题。
- 新增知识点时优先放入中文目录，例如 `高等数学/`、`线性代数/`、`概率论/`、`方法库/`、`错题/`、`题库/`。
- 重要概念、方法和错题应使用 Obsidian 双向链接，例如 `[[换元积分法]]`。
- 题库文件保持结构化，优先使用 JSON，便于脚本读取和测试。
- 新增专题时，建议同时添加：
  - `高等数学/专题名.md` 或对应学科目录下的中文笔记
  - `题库/专题名.md` 或题单索引
  - `problems/专题名.json`
  - 对应的格式测试或扩展现有测试

## 题目字段

每道题至少包含以下字段：

- `id`：唯一题号
- `topic`：专题名称
- `difficulty`：难度，建议使用 `基础`、`中等`、`提高`
- `question`：题干
- `answer`：答案
- `solution`：解析
- `tags`：标签列表

## 测试要求

修改题库后，请运行：

```bash
pytest
```

如果修改了生成练习卷的逻辑，请同时运行：

```bash
python scripts/generate_quiz.py
```

## 数学公式书写规范

本仓库采用 Obsidian 优先、兼容 GitHub 的数学公式渲染规则。修改任何 Markdown、题库或生成脚本时，都应确保公式在 Obsidian 和 GitHub 中都能稳定阅读。

- 禁止把 Markdown、Python、JSON 文件压缩成单行或少量超长行。
- GitHub Raw 文件中的真实换行必须正确，不能依赖编辑器或网页的视觉自动换行。
- Markdown 标题、段落、列表项和公式块都要使用真实换行分隔。
- Python 文件必须保持正常缩进，import、函数定义、循环、条件和 return 等语句应各自独立成行。
- JSON 文件使用 2 个空格缩进格式化，保持字段结构清晰。
- 长公式必须使用行间公式，并严格写成：

```markdown
$$
\int_a^b f(x)\mathrm{d}x
$$
```

- 行间公式的开始分隔符和结束分隔符必须分别单独占一行，前后保留空行。
- 不要把行间公式写在有序列表或无序列表的同一行。
- 尽量避免在列表内部嵌套复杂行间公式；需要编号说明时，优先使用 `**1. 标题：**` 这样的加粗编号标题。
- 行内公式只用于短表达式，例如 `$f(x)$`、`$a \ne -1$`、`$m \le f(x) \le M$`。
- 积分微分统一使用 `\mathrm{d}x`、`\mathrm{d}t`、`\mathrm{d}u`，不要使用窄空格命令或逗号放在微分记号前的写法。
- 不要使用标记为 text 的代码块展示数学公式。
- JSON 题库中的公式也要使用 LaTeX，并正确转义反斜杠，例如 `"计算 $\\int_0^1 x^2\\mathrm{d}x$"`。
- 修改 `scripts/generate_quiz.py` 时，要确保生成的 `quiz.md` 中长公式独立成块：

```markdown
$$
\int_0^1 x^2\mathrm{d}x
$$
```

- 块级公式中不要把等号、加号、减号、不等号或比较符号单独放一行，避免 GitHub 把它们误判为 Markdown 标记。
- 多行等式或多行不等式统一使用 `aligned` 环境，避免 `=` 单独占行，例如：

```markdown
$$
\begin{aligned}
\int_a^b f(x)\mathrm{d}x
&= \int_a^c f(x)\mathrm{d}x \\
&\quad + \int_c^b f(x)\mathrm{d}x
\end{aligned}
$$
```

## Git 约定

- 不修改 Git 配置。
- 不提交临时文件、缓存文件或与复习系统无关的产物。
- 保持提交内容聚焦，避免无关重构。
